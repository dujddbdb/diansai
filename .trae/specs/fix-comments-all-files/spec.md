# 智能车项目代码注释规范修复 - 产品需求文档

## Overview
- **Summary**: 对三个项目（car智能车、eye视觉云台、k230视觉检测）的所有业务代码文件进行注释规范修复，包括头文件详细注释、源文件逻辑块注释、底层函数逐行注释、宏定义和参数调优说明，统一使用中文注释，清理冗余的头文件引用注释，最终提交到git仓库。
- **Purpose**: 提高代码可维护性，方便后续调试和参数调优，确保开发者能够快速理解代码逻辑和参数含义。
- **Target Users**: 智能车开发团队成员、代码维护人员

## Goals
- 头文件中所有函数声明、全局变量、宏定义、结构体都有清晰的中文注释
- 源文件中函数内部逻辑块有明确的注释说明
- 控制底层函数有逐行简单注释，便于维护
- 需要反复调整的参数（PID、滤波系数等）有调大调小效果说明
- 清理冗余的头文件引用注释，保持代码简洁
- 所有注释统一使用中文
- 覆盖三个项目所有业务代码文件，不遗漏

## Non-Goals (Out of Scope)
- 不修改 STM32 标准库驱动代码（libraries/ 目录下的 CMSIS 和 StdPeriph_Driver）
- 不修改 OLED 字库文件（oleddfont.h、bmp.h 等纯数据文件）
- 不改变代码逻辑，只添加/修改注释
- 不重构代码结构
- 不添加新功能

## 项目范围
共三个项目：
1. **project_1.0 car/** - 智能车巡线控制系统（C语言，STM32F4）
2. **project_1.0 eye/** - 视觉云台追踪系统（C语言，STM32F4）
3. **k230/** - K230视觉检测脚本（Python）

## Background & Context
- car项目：智能车巡线控制，包含灰度传感器、电机驱动、陀螺仪、PID控制、直角检测等
- eye项目：视觉云台追踪，包含步进电机、视觉处理、IMU补偿、激光触发等
- k230项目：K230端视觉检测算法，包含矩形检测、ROI追踪、透视变换等
- 当前代码注释存在以下问题：
  - 部分文件头文件引用有大量冗余注释
  - 部分函数内部逻辑块缺少注释
  - 底层函数缺少逐行注释
  - 宏定义和可调参数缺少调优说明
  - 注释风格不统一

## Functional Requirements
- **FR-1**: 头文件注释 - 所有头文件中的函数声明、全局变量、结构体、宏定义都要有详细中文注释
- **FR-2**: 源文件逻辑块注释 - 函数内部按逻辑块添加中文注释，说明每个逻辑块的功能
- **FR-3**: 底层函数逐行注释 - 控制底层函数（如寄存器操作、硬件初始化等）有逐行简单注释
- **FR-4**: 可调参数说明 - PID参数、滤波系数、阈值等需要反复调整的参数，注释说明调大调小的效果
- **FR-5**: 清理冗余注释 - 移除头文件引用处的冗余/重复注释
- **FR-6**: 中文注释统一 - 所有注释使用中文，风格一致
- **FR-7**: 全文件覆盖 - 覆盖三个项目所有业务代码文件，不遗漏

## Non-Functional Requirements
- **NFR-1**: 注释清晰准确 - 注释要准确反映代码功能，不误导
- **NFR-2**: 注释简洁 - 避免冗余注释，在必要处提供有价值的信息
- **NFR-3**: 代码可编译 - 修改注释后C代码仍能正常编译

## Constraints
- **Technical**: C语言 + Python，STM32F4平台 + K230平台
- **Business**: 不改变代码逻辑，只修改注释
- **Dependencies**: 无外部依赖

## Assumptions
- "驱动底层"指 STM32 标准库和纯硬件寄存器操作封装，这部分不修改
- "控制底层"指 bsp 层的硬件驱动封装（如tb6612、encoder、grayscale等），需要添加注释
- "控制逻辑"指 track.c/h、vision.c/h 等核心控制算法，需要详细注释
- "使用层"指 main.c 等应用层代码，需要注释
- k230 Python 项目按相同原则处理

## Acceptance Criteria

### AC-1: 头文件注释完整性
- **Given**: 任意业务代码头文件
- **When**: 检查其中的函数声明、全局变量、结构体、宏定义
- **Then**: 每个元素都有清晰的中文注释说明其功能/用途
- **Verification**: `human-judgment`

### AC-2: 源文件逻辑块注释
- **Given**: 任意业务代码源文件
- **When**: 检查函数内部逻辑
- **Then**: 每个相对独立的逻辑块前都有中文注释说明该块的功能
- **Verification**: `human-judgment`

### AC-3: 底层函数逐行注释
- **Given**: 控制底层函数（硬件初始化、寄存器配置等）
- **When**: 检查函数实现
- **Then**: 关键行有逐行简单注释，说明该行操作的含义
- **Verification**: `human-judgment`

### AC-4: 可调参数调优说明
- **Given**: 配置文件中的可调参数（如 track_config.h、vision_config.h、k230 main.py 参数区）
- **When**: 检查 PID参数、滤波系数、阈值等参数
- **Then**: 参数注释说明调大/调小的效果或影响
- **Verification**: `human-judgment`

### AC-5: 冗余头文件引用注释已清理
- **Given**: main.c 等源文件的头文件引用部分
- **When**: 检查 #include 语句
- **Then**: 没有大段冗余的头文件功能说明注释，只保留必要的简短说明或无注释
- **Verification**: `human-judgment`

### AC-6: 中文注释统一
- **Given**: 所有修改的文件
- **When**: 检查注释语言
- **Then**: 所有注释使用中文
- **Verification**: `human-judgment`

### AC-7: 全文件覆盖
- **Given**: 三个项目的业务代码文件列表
- **When**: 对比修改前后的文件列表
- **Then**: 所有需要修改的文件都已处理，没有遗漏
- **Verification**: `programmatic`

### AC-8: 代码可编译
- **Given**: 修改后的C代码
- **When**: 编译项目
- **Then**: 编译通过，无错误
- **Verification**: `programmatic`

### AC-9: 已提交到git
- **Given**: 所有注释修改完成
- **When**: 检查git状态
- **Then**: 代码已提交并推送到远程仓库
- **Verification**: `programmatic`
